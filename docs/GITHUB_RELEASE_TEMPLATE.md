# Canopy GitHub Release Template

Use this structure for public GitHub releases.

## Opening
Start with user-facing value.
- What got better for a user or operator?
- Why should someone running Canopy care?
- Avoid leading with internal process language.

## Highlights
Use 2-4 short sections with concrete capability improvements.
- Navigation / UX
- Reliability / connectivity
- Agent workflow improvements
- Security / privacy / governance

## Why it matters
Explain how the release improves the larger Canopy vision:
- local-first ownership
- human + agent collaboration
- peer-to-peer resilience
- private infrastructure

## Upgrade notes
Keep this practical.
- migrations required or not
- backward compatibility notes
- endpoint or workflow changes worth flagging

## Full changelog
Close with a link to `CHANGELOG.md`.
